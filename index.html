intro to VCS

Version Control Systems (VCS)

  • wise programmers use it in code development
  • track and provide control over changes of data
  • helps collaboration!
  • allows storing versions
  • restoring a previous version
  • compare changes over time
  • understand what happened
  • see who to blame for a bug
  • consistency & integrity of data
  • backup your work

history

Before-VCS epoch

  • prehistory: Keep your own versions
  • stone-age: Use diff, emails and tars

Centralized VCS (CVCS) epoch

  • Bronze age: CVS, a wrong intro
  • Iron age: SVN, perforce

Distributed VCS (DVCS) epoch

  • Modern era: bazaar, arch, monotone, darcs, bitkeeper

Linus and the war of (D)VCS

  • Linus was not happy with any of the above!
  • He stepped out of Linux development for a month to create his second child, git
  • mercurial, another child of the war
  • git wins the war, becomes the standard in Open Source community, followed by mercurial

DVCS vs CVCS

Centralized

  • Only a single copy of the repository (unsafe)
  • Communication with the server for each operation (slow)
  • Server is offline or corrupted; no job done (offline cost)
  • Everyone has access to the repository (no access control)
  • Everything is stored to the server; even failed things (redundacy)
  • Pros: Rather simple interaction (not flexible though)

Distributed

  • More powerful and flexible workflows (promote collaboration)
  • Each user has its own repository (safety)
  • Operations are done locally (performance)
  • Branches is the way to go (easy to create, delete and merge)
  • No need to give write access control (let them read but not write)
  • Cons: Initial checkout is more costly (download the whole history)
  • Cons: A little more difficult to grasp UI (interaction overhead)

3 trees + add

three trees

  • working directory → sandbox, holds the files
  • index → snapshot of changes proposed for next revision
  • HEAD → pointer to last snapshot
  • HEAD (usually) points where the branch points

create our first file

  • master Current active branch (if no active branch → detached mode)
  • $ touch first
  • File added to working directory

git add [<file>]

  • Add content to the staging index
  • $ git add first
  • File added to index

commit & log

git commit [<file>]

  • Stores the contents of the index in a new commit object
  • along with who, when, and why made those changes
  • parent of commit object is the previous commit
  • Creates a SHA-1 20 byte hash using the above
  • HEAD and master point to new commit
  • $ git commit first
  • Add & commit the file at the same time
  • $ git commit -a first

git log

  • $ git log --decorate --graph
    * commit 8484f4fccb80e625b7c05cdc214014a32d176777 (HEAD)
    | Author: Panagiotis Papadakos <papadako@gmail.com>
    | Date: Tue Oct 20 01:15:40 2015 +0300
    |   Add some content

branching

git branch <name>

  • Branching is a very cheap and widely used operation
  • Develop features/correct bugs in isolation
  • Just creates a new pointer where HEAD points
  • $ git branch featureX
  • Only one activeBranch (the rest notActiveBranches)
  • master is the default branch (commit still updates the master pointer)

git checkout [<name>]

  • Switch to another branch
  • Updates files in the working tree to match the version
        in the index or the specified tree
  • <name> a branch name or commit object (if not given, HEAD)
  • If modifications cannot be applied cleanly
       ...git won't switch to another branch (git commit or git stash)
  • $ git checkout featureX

merging

git merge <commit>

  • Easy branching without easy merging has no value!
  • Join two or more development histories (branches) together
  • Example: Made a number of commits in branch featureX
  • Incorporate them in master by:
         git checkout master
         git merge featureX
  • The simplest of all merges: fast forward !
  • git has a lot of merging policies

merging Part β

git merge <commit>

  • A more difficult case!
  • Make a number of commits to featureX
  • Then go to master
  • Make a number of commits
  • Merge master with featureX
  • A commit object can have many parents
  • Even if we can fast forward...
       ... git merge -no-ff will create a new commit

conflicts

Merging can lead to Conflicts

  • git tries hard and is smart enough in avoiding them
  • But when conflicts occur, no need to be scared!
  • Let them fight and choose the winner
        i.e. review the code and resolve them

    • <<<<<<< HEAD
    I am the bug!
    =======
    No, I am the bug!
    >>>>>>> master
    Resolve bug

tips

some guidelines

  • git tracks content not files
  • be careful with git mv (commit both the old and new file)
  • commit often (makes easier code reviewing and bug tracking)
  • merge master often (reduces conflicts)
  • keep binary files in another repository (overhead per each clone)
  • delete branches that you do not need (branch overhead)
  • be careful about formatting
  • → useful changes are hidden in noise
  • → produce conflicts
  • → make explicit commits about formatting
